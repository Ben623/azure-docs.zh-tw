# <a name="azure-security-policies-monitored-by-security-center"></a>資訊安全中心監視的 azure 安全性原則
本文提供一份 Azure 原則定義，您可以在 Azure 資訊安全中心中監視。 如需有關安全性原則的詳細資訊，請參閱[搭配使用安全性原則](tutorial-security-policy.md)。

## <a name="available-security-policies"></a>可用的安全性原則

若要深入了解資訊安全中心會監視內建原則，請參閱下表：

| 原則 | 原則的功能 |
| --- | --- |
|應該啟用診斷記錄檔中的虛擬機器擴展集|我們建議您啟用記錄檔，以便的活動記錄可供事件或洩露之後進行調查。|
|要從事件中樞命名空間移除 RootManageSharedAccessKey 以外的所有授權規則|Azure 事件中樞用戶端不應該使用可存取所有佇列和主題在命名空間中的命名空間層級存取原則。 若要對齊的最低權限的安全性模型，您應該建立存取原則，佇列和主題，以提供特定的實體存取權的實體層級。|
|您應該定義事件中樞實體上的授權規則|稽核授權規則授與權限的最低存取權的事件中樞實體上的存在。|
|具有防火牆和虛擬網路組態的儲存體帳戶的存取權應該限制|稽核儲存體帳戶防火牆設定中不受限制的網路存取。 因此，只有來自允許網路的應用程式可以存取儲存體帳戶，請設定網路規則。 若要允許來自特定網際網路或內部部署用戶端的連線，授與存取流量來自特定的 Azure 虛擬網路或公用網際網路 IP 位址範圍。|
|稽核自訂 RBAC 規則的使用方式|稽核內建的角色，例如 「 擁有者、 參與者、 讀取者 」，而不是自訂的角色型存取控制 (RBAC) 角色，也就是容易發生錯誤。 使用自訂角色會被視為例外狀況，並需要嚴格的審查和威脅分析模型。|
|應該啟用 Azure Stream Analytics 中的診斷記錄檔|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用安全傳輸至儲存體帳戶|稽核儲存體帳戶中的安全傳輸的需求。 安全傳輸這個選項會強制您的儲存體帳戶僅接受來自安全連線 (HTTPS) 的要求。 使用 HTTPS 可確保伺服器和服務之間的驗證。 它也會保護傳輸中的資料從網路層攻擊，例如-攔截，竊聽，及工作階段攔截。|
|SQL server 的 azure AD 系統管理員應該將他們佈建|若要啟用 Azure AD 驗證的 SQL Server 的 Azure Active Directory (Azure AD) 系統管理員佈建的稽核。 Azure AD 驗證支援簡化的權限管理和集中式身分識別管理的資料庫使用者和其他 Microsoft 服務。|
|應該從服務匯流排命名空間移除 RootManageSharedAccessKey 以外的所有授權規則|Azure 服務匯流排用戶端不應該使用可存取所有佇列和主題在命名空間中的命名空間層級存取原則。 若要對齊的最低權限的安全性模型，建立佇列和主題，以提供特定的實體存取權的實體層級的存取原則。|
|應該啟用服務匯流排中的診斷記錄檔|稽核記錄檔啟用並將其註冊，以一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該設定為 EncryptAndSign Service Fabric 中的 ClusterProtectionLevel 屬性|Service Fabric 提供三個節點間通訊使用主要叢集憑證的保護層級：無、 簽署及 EncryptAndSign。 設定保護層級可確保所有節點對節點的訊息皆經過加密及數位簽署。|
|用戶端驗證應使用 Azure Active Directory|稽核使用只能透過 Service Fabric 中的 Azure AD 的用戶端驗證。|
|服務應該已啟用的搜尋中的診斷記錄|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用只安全連線至 Redis 快取|稽核僅啟用透過 Azure 快取的 SSL 適用於 Redis 的連線。 使用安全的連接可確保伺服器和服務之間的驗證。 它也會保護傳輸中的資料從網路層攻擊，例如-攔截，竊聽，及工作階段攔截。|
|在 Logic Apps 中的診斷記錄檔應該已啟用|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用金鑰保存庫中的診斷記錄檔|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用事件中樞中的診斷記錄檔|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用 Azure Data Lake Store 中的診斷記錄檔|稽核記錄檔啟用，並讓它們保持最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用 Data Lake Analytics 中的診斷記錄檔|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|儲存體帳戶應該移轉到新的 AzureRM 資源|使用 Azure Resource Manager 儲存體帳戶提供安全性增強功能。 它們包括： <br>-更強的存取控制 (RBAC)<br>-更好的稽核<br>-Azure Resource Manager 為基礎的部署和管理<br>若要管理的身分識別存取<br>存取 Azure Key vault 祕密<br>-Azure AD 為基礎的驗證<br>-支援標記和資源群組，以簡化安全性管理|
|虛擬機器應該移轉至新的 AzureRM 資源|使用您的虛擬機器的 Azure Resource Manager 提供的安全性增強功能。  它們包括： <br>-更強的存取控制 (RBAC)<br>-更好的稽核<br>-Azure Resource Manager 為基礎的部署和管理<br>若要管理的身分識別存取<br>存取 Azure Key vault 祕密<br>-Azure AD 為基礎的驗證<br>-支援標記和資源群組，以簡化安全性管理|
|應該在 Batch 帳戶上設定計量警示規則|在 Azure Batch 帳戶，以啟用所需的度量的計量警示規則的稽核設定。|
|應該啟用 Batch 帳戶中的診斷記錄|稽核記錄檔啟用，並保持為最多一年。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該在自動化帳戶的變數上啟用加密|請務必儲存機密資料時，啟用加密的 Azure 自動化帳戶中的變數資產。|
|應該啟用在 App Service 中的診斷記錄檔|稽核應用程式中診斷記錄的啟用情形。 發生安全性事件，或您的網路遭到入侵時，這會建立活動記錄進行調查。|
|應該啟用 SQL database 上的透明資料加密|稽核 SQL 資料庫的透明資料加密狀態。|
|應該啟用 SQL server 稽核|稽核存在 SQL 稽核伺服器層級。|
|\[預覽]:在 Azure 資訊安全中心中監視未加密的 SQL 資料庫|Azure 資訊安全中心監視未加密的 SQL 伺服器或資料庫的建議。|
|\[預覽]:在 Azure 資訊安全中心中監視未稽核的 SQL 資料庫|Azure 資訊安全中心會監視 SQL server 並沒有建議的方式開啟 SQL 稽核功能的資料庫。|
|\[預覽]:應該在機器上安裝系統更新|Azure 資訊安全中心會監視您伺服器上遺漏的安全性系統更新建議。|
|\[預覽]:稽核 Blob 未加密的儲存體帳戶|稽核未使用 blob 加密的儲存體帳戶。 這只適用於 Microsoft 儲存體資源類型，不來自其他提供者的存放裝置。 建議的方式，azure 資訊安全中心會監視可能的網路中即時存取。|
|\[預覽]:在 [Just-in-time 網路存取控制應該套用在虛擬機器上|建議的方式，azure 資訊安全中心會監視可能的網路中即時存取。|
|\[預覽]:應該在虛擬機器上啟用自適性應用程式控制|Azure 資訊安全中心會監視可能的應用程式允許清單設定。|
|\[預覽]:應設定遺漏的虛擬機器的網路安全性群組|Azure 資訊安全中心監視的網路安全性群組擁有太寬鬆的規則，建議的方式。|
|\[預覽]:應該修復您的電腦上的安全性設定中的弱點|Azure 資訊安全中心監視建議的方式不符合設定的基準的伺服器。| 
|\[預覽]:應該在虛擬機器上安裝 endpoint protection|Azure 資訊安全中心監視沒有安裝的 Microsoft System Center Endpoint Protection 代理程式，建議的伺服器。|
|\[預覽]:應該在虛擬機器上套用磁碟加密|Azure 資訊安全中心監視不需要所建議來啟用磁碟加密的虛擬機器。|
|\[預覽]:應該修復弱點的弱點評量解決方案|監視的弱點評量解決方案和弱點評量解決方案中沒有 Azure 資訊安全中心建議的 Vm 所偵測到的弱點。|
|\[預覽]:在 Azure 資訊安全中心中監視未受保護的 Web 應用程式|Azure 資訊安全中心監視 web 應用程式沒有 web 應用程式防火牆保護的建議。|
|\[預覽]:應該在虛擬機器上安裝端點保護解決方案|Azure 資訊安全中心監視的網路沒有下一個層代防火牆保護，建議的端點。|
|\[預覽]:應該修復您的 SQL database 中的弱點|監視弱點評定掃描結果，並建議修復的資料庫弱點。|
|\[預覽]:應獲指定最多 3 個擁有者的訂用帳戶|我們建議您指定最多三個訂用帳戶擁有者，可以降低遭入侵的擁有者的破壞的可能性。|
|\[預覽]:應該有一個以上的擁有者指派給您的訂用帳戶|我們建議您指定一個以上的訂用帳戶擁有者，以確保系統管理員存取備援。|
|\[預覽]:應該與您的訂用帳戶的擁有者權限帳戶啟用 MFA |應該啟用多重要素驗證 (MFA)，以避免出現帳戶或資源缺口的擁有者權限的所有訂用帳戶。|

|\[預覽]:在您的訂用帳戶，具有讀取權限均應啟用 MFA|應該具有讀取權限，以避免出現帳戶或資源缺口的所有訂用帳戶啟用多重要素驗證。|
|\[預覽]:應該從您訂用帳戶中移除具有擁有者權限已被取代的帳戶|從您的訂用帳戶，應該移除不再使用的帳戶的擁有者權限。 已封鎖不再使用的帳戶登入。|

|\[預覽]:應該從您的訂用帳戶中移除具有擁有者權限的外部帳戶|應該移除擁有者權限的外部帳戶，從您的訂用帳戶，以防止權限的存取。|
